On a toujours écrit des specs.

Mais un **humain comblait les trous** avec son contexte implicite, une conversation de couloir, son expérience dans un domaine, sa compréhension d’un métier, son bon sens.

**Un agent IA ne comble rien, il amplifie ce qu'il reçoit**.

Des specs floues donnaient des tickets mal estimés.

Aujourd'hui si on ne prend pas le temps de clarifier des specs pour un agent IA, elles donnent des centaines de lignes de code dans la mauvaise direction.

Depuis 2025, une vague d’outils (Kiro, SpecKit, OpenSpec…) a remis le Spec-Driven Development sous le feu des projecteurs.

Ce n’est pas une nouveauté,  c’est une exigence devenue visible.

Dans ce talk, je présente la philosophie d’OpenSpec en illustrant avec un workflow en 4 étapes (Proposal, Design, Specs, Tasks) que nous utilisons depuis plusieurs mois dans mon équipe (6 devs, POs,PM,).

Vous repartirez avec:

-  un workflow concret et adaptable à votre contexte
- Les signaux d'alerte qui indiquent qu'une spec va générer du code inutilisable avant même de lancer un agent sur l’implémentation.
- une stratégie d’adoption progressive pour introduire cette rigueur sans friction dans une équipe existante.

On parlera aussi des impacts humains, parce que le Spec Driven Development, c’est autant un changement de culture PM/PO/DEV qu’un changement d’outil.
